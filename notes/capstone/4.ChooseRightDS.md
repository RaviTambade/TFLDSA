This is one of the most important lessons in software engineering: **the business problem determines the data structure**. Teach students to think like architects, not just programmers.

| Business Problem                   | Business Requirement                      | Best Data Structure          | Why?                                                  |
| ---------------------------------- | ----------------------------------------- | ---------------------------- | ----------------------------------------------------- |
| Register one customer              | Keep all customer details together        | **Structure**                | Groups related fields into one record.                |
| Store 10,000 customers             | Store and access many customer records    | **Array of Structures**      | Simple, indexed storage with fast access.             |
| Calculate insurance premium        | Premium depends on Age × Policy Type      | **2D Array (Matrix)**        | Represents rows and columns naturally.                |
| Register insurance claims          | Claims arrive every day; count is unknown | **Linked List**              | Dynamic size; easy insertion and deletion.            |
| Process claims                     | Process in arrival order                  | **Queue**                    | First-In, First-Out (FIFO) ensures fairness.          |
| Handle emergency claims            | Critical claims first                     | **Priority Queue**           | Highest-priority claim is processed first.            |
| Undo accidental changes            | Restore the last modification             | **Stack**                    | Last-In, First-Out (LIFO) supports Undo.              |
| Search policy by Policy ID         | Millions of policies need fast search     | **Binary Search Tree (BST)** | Faster searching than linear scanning when organized. |
| Customer login                     | Find user by email quickly                | **Hash Table**               | Near constant-time lookup using a hash function.      |
| Find highest premium or top claims | Quickly retrieve maximum values           | **Heap (Max Heap)**          | Efficient access to the largest element.              |
| Connect branch offices             | Represent cities and roads                | **Graph**                    | Models interconnected locations and routes.           |

## Enterprise Software Architect's Thinking

Instead of asking:

> **"Which data structure should I use?"**

Teach students to ask:

1. **What is the business problem?**
2. **What operations are performed most often?**
3. **Is the data fixed or dynamic?**
4. **Do I need fast searching, insertion, deletion, or ordering?**
5. **Which data structure solves these requirements efficiently?**

### Example

**Business Requirement:**

> "Insurance claims arrive throughout the day, and we don't know how many there will be."

**Analysis:**

* Dynamic number of records ✔
* Frequent insertion ✔
* Frequent deletion ✔

**Choice:** **Linked List**

---

**Business Requirement:**

> "Claims must be processed in the same order they are received."

**Analysis:**

* First claim should be processed first ✔

**Choice:** **Queue**

---

**Business Requirement:**

> "Accident and medical emergency claims must be handled before normal claims."

**Analysis:**

* Processing depends on importance, not arrival time ✔

**Choice:** **Priority Queue**

---

## Software Engineering Principle

```text
Business Requirement
        ↓
Requirement Analysis
        ↓
Choose Appropriate Data Structure
        ↓
Design Algorithm
        ↓
Implement Solution
        ↓
Test with Real Business Data
```

This is exactly how professional software engineers design systems: they **analyze the business requirement first, then choose the data structure that best supports the required operations**, rather than trying to force every problem to fit the same data structure.
