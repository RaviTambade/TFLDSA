### ⚙️ **When an Application Runs → It Becomes a Process**

Now imagine one actor is performing — they go from backstage to center stage.

The OS gives it:

| Memory Type      | Purpose                                                            |

* Reads instructions from code section.
* Takes input from stack and heap.
* Talks to the CPU to execute line by line.

Every process has:
- ✅ At least **1 Main Thread**
- ✅ Can have **multiple threads** (like multiple cooks in a kitchen, working together)

## 🛡️ **Chapter 2: Process Scheduling, CPU Registers & Interrupts**

The king of this computer kingdom is the **CPU** — swift, powerful, and always ready to execute the next command. But the king can only focus on **one citizen (process)** at a time. Yet thousands of citizens (processes) say, “Execute me first!”

So… how does the king manage everyone **fairly and efficiently**?


### 👑 **1. The Royal Clock: CPU Scheduling**

Imagine a courtroom with many people standing in line for justice (execution). The CPU uses **scheduling algorithms** to decide who goes next.

| Algorithm                          | Story Version                                   | Real Version                                            |
| ---------------------------------- | ----------------------------------------------- | ------------------------------------------------------- |
| **FCFS (First Come, First Serve)** | First person in line is heard                   | Processes executed in arrival order                     |
| **Shortest Job First (SJF)**       | Quick tasks go first                            | Process with least execution time                       |
| **Round Robin (RR)**               | Each person gets 10 seconds to speak; then next | Every process gets a time slice (quantum)               |
| **Priority Scheduling**            | King hears nobles first, farmers later          | Higher priority processes go earlier                    |
| **Multilevel Queue**               | VIP line, normal line, peasant line             | Processes in separate queues: system, user, batch, etc. |

The **scheduler** is like the king’s secretary—it decides **who gets the CPU next** and for how long.

### 🧠 **2. CPU Registers – The King’s Memory Pockets**

While listening to a process, the king keeps essential information in **CPU registers**, like:

| Register Type                                      | What It Stores                                | Story Interpretation                       |
| -------------------------------------------------- | --------------------------------------------- | ------------------------------------------ |
| **Program Counter (PC)**                           | Address of next instruction                   | “What should I do next?”                   |
| **Stack Pointer (SP)**                             | Top of the stack                              | “Where is my checklist of function calls?” |
| **Accumulator / General Registers (R0, R1, etc.)** | Temporary data/calculations                   | "Let me store this number quickly!"        |
| **Flags / Status Register**                        | Results of operations (Zero, Carry, Negative) | “Was the last action a success?”           |


### ⏳ **3. Context Switching – When King Changes His Focus**

Suddenly, another process shouts: **“Urgent! I need the CPU!”**

So the current process says, “Okay, I’ll wait”. The OS performs **context switching**, which means:

- ✔ Save current king’s thoughts (register values, PC, stack) on **PCB (Process Control Block)**
- ✔ Load new process’s saved state into the registers
- ✔ Start executing new process

This is like the king putting one case file back into the drawer and picking another one.


### ⚡ **4. Interrupts – When Someone Knocks on the King’s Door**

While the king is judging a case, a soldier runs in:

> “Your Majesty! The keyboard has been pressed!”

This event is called an **Interrupt**.

Types of Interrupts:

| Type                   | Example                            | Story Meaning                                     |
| ---------------------- | ---------------------------------- | ------------------------------------------------- |
| **Hardware Interrupt** | Keyboard press, mouse click, timer | A soldier delivering an urgent real-world message |
| **Software Interrupt** | System calls (`printf`, `read`)    | A citizen politely requests kingdom services      |
| **Timer Interrupt**    | Time slice over in Round Robin     | Royal clock rings: “Next citizen’s turn!”         |

When an interrupt occurs:

1. CPU pauses current process
2. Saves the context to memory (like marking the page in a book)
3. Jumps to **Interrupt Service Routine (ISR)** – a special function to handle the problem
4. After solving, CPU resumes the original process


### 🗂️ **5. Putting It All Together**

- ✅ **Stack** – Call history and local variables
- ✅ **Heap** – Dynamically allocated objects
- ✅ **Data & Code Sections** – Program's fixed layout
- ✅ **Scheduler** – Decides which process speaks
- ✅ **CPU Registers** – King’s working memory
- ✅ **Interrupts** – Emergency messages
- ✅ **Context Switch** – Save & load king’s focus


# 🧵 **Chapter 3: Threads – Multiple Workers in One Kingdom**

In the previous chapter, we met **processes** — like independent citizens in the kingdom. Each had their own:

* 🧠 **Private memory (Stack, Heap, Code, Data)**
* 📂 **Own files, variables, registers**
* 👮 **Protected area from other processes**

But soon, the king (CPU) discovered a problem…

> "Why should a single process citizen do everything alone? What if he had multiple workers to help him?"

Enter **Threads** — lightweight helpers inside one process.

## 👨‍👨‍👦 **1. Process vs Thread – Story Style**

| Feature       | Process (Citizen)                                                 | Thread (Citizen's Worker)                                    |
| ------------- | ----------------------------------------------------------------- | ------------------------------------------------------------ |
| Memory        | Has its own house (Stack, Heap, Data, Code)                       | Lives inside the same house                                  |
| Isolation     | Other processes can't access its home                             | Threads inside one process share everything                  |
| Communication | Slow (letters → Interprocess Communication: pipes, shared memory) | Fast (they talk directly using shared memory)                |
| Creation Cost | Heavy (like giving land, home, address)                           | Light (just adding helpers in same home)                     |
| Crash Impact  | One citizen dying doesn’t affect others                           | If one worker corrupts shared memory → whole house collapses |

## 🏠 **2. What Do Threads Share? What Is Separate?**

Inside a single process, all threads **share**:

| Shared Among Threads                | Not Shared (Each Thread Has Its Own)    |
| ----------------------------------- | --------------------------------------- |
| Code section (program instructions) | Program Counter (PC)                    |
| Data section (global/static vars)   | CPU Registers                           |
| Heap (dynamic memory)               | Stack (function calls, local variables) |
| Open files / resources              | Thread ID (TID)                         |

🎯 **So, each thread has its own stack but shares the heap with siblings.**

## 🧠 **3. Real-Life Analogy**

Imagine a **restaurant** (process). The restaurant has:

* One kitchen (heap)
* One recipe book (code)
* One storage room of ingredients (data)
* Multiple chefs (threads), each with:

  * Their own notepad (stack)
  * Their own cooking steps (program counter)

If one chef writes a wrong recipe in the shared book → all chefs suffer!

## ⚙️ **4. How Threads Work in the OS**

A thread still needs OS support. So the OS manages:

* **Thread ID (TID)**
* **Registers & Stack pointer**
* **State** (Running, Ready, Blocked)
* **Shared PCB of the parent process**

When switching between threads of the same process:
- ✅ Fast context switch (no change of memory address space)
When switching between different processes:
- ❌ Slow — OS must change memory maps, PCB, etc.

## 💻 **5. Example Code – Thread in Python**

```python
import threading

def task(name):
    print(f"Thread {name} is running")

# Creating threads
t1 = threading.Thread(target=task, args=("A",))
t2 = threading.Thread(target=task, args=("B",))

t1.start()
t2.start()

t1.join()
t2.join()
```

🧵 Here, **Thread A** and **Thread B** run inside the **same process**, sharing memory.

## ⚠️ **6. Dangers – Race Condition & Deadlock**

### 🏁 Race Condition:

Two or more threads trying to **modify the same data at the same time**.

Example:

```
wallet = 100

Thread1: read wallet (100) → add 50 → write 150
Thread2: read wallet (100) → add 20 → write 120  ❌ (lost update)
```

### 🗝️ Solution: Use Locks (Mutex)

```python
import threading

wallet = 100
lock = threading.Lock()

def add_money(amount):
    global wallet
    with lock:
        wallet += amount

# Two threads adding money safely
```

### 🔒 Deadlock:

Two threads each **hold one resource and wait for the other’s resource**, like:

| Chef A          | Chef B              |
| --------------- | ------------------- |
| Holds **Knife** | Holds **Pan**       |
| Waiting for Pan | Waiting for Knife   |
| Nobody moves    | 😢 Restaurant stops |

Solution:
- ✔ Lock in same order
- ✔ Use timeout or deadlock detection

## The Visiting Card & Memory Story

*"Imagine… You go to a big Tech Exhibition or Career Fair."*

* First stall – one marketing manager gives you a **visiting card**. You hold it in your palm.
* Second stall – another gives a card, you place it **above the first one**.
* You keep visiting… 3rd stall, 4th stall, 5th… up to **100 stalls**.
* Now, your hand is full of **visiting cards placed one on top of another**.

💡 The **first card you collected is at the bottom**,
and the **last one (latest) is at the top**.

Now, someone asks:

> “Give me the 5th last card you collected.”

So what do you do?

**You start removing from the top (latest one):**

* Pop 100th card → Keep aside
* Pop 99th → aside
* Pop 98th…
* Until you get the **required card**.

That is exactly how **Stack Memory works in C, C++, Python, Java!**

## 📚 **Why Stack Exists?**

When a program runs:

* It must **keep track of which function to execute next**.
* Every time a function is called, it needs a place to store:
  - ✅ Function arguments
  - ✅ Local variables
  - ✅ Return address (where to go back after execution)

So the computer uses a memory shelf called **Stack**.

```
Stack Memory = A bundle of visiting cards kept one over another.
```

* **Push()** → When you visit a new stall (or a function is called)
* **Pop()** → When function ends or you go backwards

This principle is called **LIFO – Last In, First Out**
(last card kept = first to be removed)

## 🧠 **Then Why Do We Need Dynamic Memory (Heap)?**

Now you ask the students:

*"What if visiting cards don’t fit in your palm anymore? What if you need a bigger storage like a bag, a pouch, or a drawer?"*

Exactly!

Stack has limitations:

| Property       | Stack                   | Heap (Dynamic Memory)     |
| -------------- | ----------------------- | ------------------------- |
| Size           | Limited (small)         | Large (depends on RAM)    |
| Speed          | Very Fast               | Slower than stack         |
| Memory Control | Automatically managed   | Manually allocated/free() |
| Lifetime       | Ends when function ends | Continues until freed     |

So for **big data, unknown size, long lifetime**, we use **Heap memory** using:

* `malloc(), calloc(), realloc(), free()` in C
* `new, delete` in C++
* `new` in Java, Python internally uses heap

## 🧩 **Simple C Example (Narrated as a Story)**

```c
#include <stdio.h>
#include <stdlib.h>

int main() {
    int cardsInHand = 5; // Stored in Stack
    
    // But now I need 1000 cards – stack is not enough!
    int* visitingCards = (int*)malloc(1000 * sizeof(int)); // This is Heap!

    if(visitingCards == NULL) {
        printf("Memory not available!\n");
        return 1;
    }

    visitingCards[0] = 101;
    visitingCards[1] = 102;
    printf("First visiting card number: %d\n", visitingCards[0]);

    free(visitingCards); // Give back memory to OS
    return 0;
}
```

## 🎯 **What Did We Just Understand?**

- ✅ Stack = Visiting cards in your hand (LIFO, function calls)
- ✅ Heap = A bag to store more cards (Dynamic, large, manual control)
- ✅ Programs need both to **track execution + store growing data**
- ✅ Without stack – program won't know which function to execute next
- ✅ Without heap – program crashes when data grows beyond stack


### 🎓 ** What Really Happens When You Run a C Program?**

**Imagine a drama stage.**
The show is your program.
The actors are functions (like `main()`, `printf()`, `showDetails()`).
And the manager who controls who should come on stage and when—is the **Thread** (specifically, the *Primary Thread*).



### 🧵 **Step 1: Thread is Born**

When you double-click your `.exe` or run your C program, the **operating system creates a process** and gives birth to its first baby:

👉 **Primary Thread** – This thread’s first job is to start the drama by calling the **entry point** of the program.

But wait—**who is the entry point?**

Yes! It’s the **`main()` function**.

So, the *primary thread pushes the address of `main()` into a special memory area called the **Stack**.*


### 🗂️ **What is Stack?**

Stack is like a pile of plates in a restaurant.

* The first plate (bottom) is `main()`.
* Everything else is placed above it (LIFO – Last In, First Out).
* It is owned by the **thread**.

This stack holds:

* Function return addresses
* Local variables
* Parameters


### ⚙️ **Step 2: Code Segment vs Stack – Where is `printf` actually?**

💡 Important truth:

> “**Main is not implementing printf. Main is just *invoking* printf.**”

You don’t have `printf` code inside your program. That already exists inside the **standard C library** (`libc`).
Its compiled machine code is stored in the **Code Segment (.text section)** of your process memory.

So, your process gets divided like this:

| Memory Section                | What it Stores                                                       |
| ----------------------------- | -------------------------------------------------------------------- |
| **Code Segment (.text)**      | Machine code of functions (`main`, `printf`, any function you wrote) |
| **Data Segment (.data/.bss)** | Global/static variables                                              |
| **Heap**                      | Dynamically allocated memory (`malloc`, `calloc`)                    |
| **Stack**                     | Addresses of functions being executed + local variables              |


### 🔄 **Function Call Journey: `main()` calls `printf()`**

When `printf("Hello")` is written inside `main()`, this is what really happens:

1. Primary thread executes `main()`, its address is at the *bottom of the stack*.
2. Inside `main()`, thread sees a function call to `printf`.
3. Thread **pushes the address of `printf` into the stack**.
4. CPU jumps to `printf`'s **implementation located in the Code Segment**.
5. `printf` runs → prints text on screen.
6. When `printf` ends → its address is **popped out** of the stack.
7. Thread returns to the next line inside `main()`.


### 🔁 **What If `printf` is Inside a Loop 10,000 Times?**

For every iteration:

* Push address of `printf`
* Execute its code from code segment
* Pop address after finish

This happens **10,000 times**:

```
Push → Execute → Pop
Push → Execute → Pop
...
```

Once the loop ends:

* Only **`main()`** is on stack.
* `main()` finishes → its address is popped too.
* Stack becomes empty → **Thread exits → Program terminates.**


### 🧠 **So What Have We Learned?**

- ✅ `printf` is not *inside* `main`. It’s just *invoked* by `main`.
- ✅ Implementation of `printf` lives in **Code Segment**, not on Stack.
- ✅ Stack stores only **function call addresses + local variables**, not logic of functions.
- ✅ Primary thread calls `main`. `main` calls others (`printf`, `scanf`, user-defined functions).
- ✅ When the stack becomes empty → program ends.



### 🌟 **One-Line Magic Answer You Can Give Anywhere:**

> “Main doesn't *contain* printf. Main only *invokes* printf. The code of printf lives in the Code Segment, and only the return address is pushed on the stack during function calls.”



### 🧠 **  Why Stack Exists and Why Main Must End Gracefully**

Imagine you’re watching a drama.
Actors are ready, the stage is set, lights on… and the first actor enters.

In a computer program, that **first actor is the `main()` function**, and the person who invites it on stage is the **primary thread**, born from the **operating system**.


### 🎭 **Act 1: OS Creates the Stage**

When you run your C program:

1. **Windows / Linux / macOS** creates a *process*.
2. Inside the process, it creates a **thread** (primary thread).
3. The thread brings a special memory bag with it called the **Stack**.
4. Thread searches for the **entry point → `main()` function**.
5. It **pushes the address of `main()` into the stack** and starts executing.

This stack exists in **every OS** because without stack, no function can be called and no program can run in a structured way.


### 🎯 **But What If `main()` Is Exited Abnormally?**

Suppose you wrote something like this:

```c
int main() {
    printf("Hello");
    return 0; // OK – normal exit ✅
}
```

But if the program crashes, or someone writes:

```c
exit(0);  // directly kills the program
```

or segmentation fault occurs,

Then—**`main()` exits abnormally**. The address of `main()` gets popped from the stack suddenly.

Result?

- ✅ **Application terminates immediately.**
- ✅ **Thread ends.**
- ✅ **Process dies.**
- ❌ **No more stack frames left.**

That’s what you meant when you said:

> "OK, main function beach mein nikal gaya toh application abnormally terminate ho jayega."


### 🧱 **Why Stack Is So Important?**

Because OS wants:

- ✔️ Proper sequence of function calls
- ✔️ Proper return to the caller
- ✔️ No confusion in execution flow

That’s why stack is **LIFO – Last In, First Out**

* Whoever goes in last, comes out first.
* `main()` goes first → stays at the bottom.
* `printf()` gets called → pushed above `main()`
* After execution → popped.


### 🔁 **Example: Repeated `printf` 10,000 Times**

```c
for(int i=0; i<10000; i++) {
    printf("Hello\n");
}
```

What happens?

* **10,000 times** → address of `printf()` pushed → code executed → popped.
* After loop ends → only `main()` remains.
* When `main()` finishes → its address is popped → **Stack is empty → Program ends.**



### 💡 **Key Lesson**

> "Stack ensures every function is called and returned properly. If stack fails or overflows, your application crashes. No stack = no execution."
