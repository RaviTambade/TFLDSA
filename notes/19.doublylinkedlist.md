
# Doubly Linked List 

Welcome to one more day  learning Linked list. Let us extend the thinking from a **Singly Linked List** to a **Doubly Linked List**. The focus is on **reverse engineering**, **first-principles thinking**, and helping students understand *why* a doubly linked list exists before writing code.
## A Transflower Mentor Session

> **"Class, today we are not learning a new data structure. We are solving a limitation in our previous design."**

Yesterday, we built a **Singly Linked List**.

It worked beautifully.

We could:

* Insert nodes
* Delete nodes
* Search nodes
* Traverse from beginning to end

But every engineer should ask one important question:

> **"What can't this design do?"**

That question is called **engineering thinking**.


# Step 1 — Review Our Previous Design

Imagine our linked list as a train.

```text
Head

 ↓

+------+-------+
| 10   |   •-----> +------+-------+
+------+-------+    | 20   |   •-----> +------+-------+
                    +------+-------+    | 30   | NULL |
                                        +------+-------+
```

Every bogie knows only one thing.

> **Who comes after me?**

It has no idea who comes before it.


## The Problem

Suppose you are standing at the last bogie.

Can you go back?

No.

You must start again from the head.

Imagine Spotify.

Imagine WhatsApp chats.

Imagine your browser history.

Imagine an image gallery.

Users expect to move:

Forward

Backward

Forward

Backward

Our singly linked list cannot do this efficiently.

So what should we add?


# Step 2 — Think Before Coding

Suppose every bogie carries two connectors.

One connector points to the next bogie.

Another connector points to the previous bogie.

Now every bogie knows:

* Who is before me?
* Who is after me?

Congratulations.

You have just invented the **Doubly Linked List**.

# Step 3 — Draw Before You Code

```text
NULL

   ↑

+---------+---------+---------+
| previous|  data   |  next   |
+---------+---------+---------+

```

A complete list now looks like this.

```text
NULL

 ↑

+----+-----+----+     +----+-----+----+     +----+-----+----+
|NULL| 10  | •------->| •  | 20  | •------->| •  | 30  |NULL|
+----+-----+----+     +----+-----+----+     +----+-----+----+
      ←--------------------←--------------------
```

Every node knows both neighbours.

That is the only difference.

One extra pointer changes everything.

# Step 4 — Design the Node

Before writing algorithms, build the smallest building block.

```cpp
class Node
{
public:

    int data;

    Node* previous;

    Node* next;

    Node(int value);
};
```

Mentor:

> "Notice something beautiful."

Earlier we had:

Data

Next

Now we have:

Previous

Data

Next

We are not replacing the design.

We are extending it.

That is how software evolves.

# Step 5 — Give Life to the Node

```cpp
Node::Node(int value)
{
    data = value;

    previous = nullptr;

    next = nullptr;
}
```

Every newly created bogie is standing alone.

It has no neighbour.

So both pointers are NULL.

# Step 6 — Rethink the Linked List Class

Our previous LinkedList looked like this.

```cpp
class LinkedList
{
private:

    Node* head;

public:

    LinkedList();

    void insert(int);

    void remove(int);

    void display();
};
```

Can we improve it?

Yes.

If we keep only the head...

We still have to travel to the last node every time we insert.

That is unnecessary.

Let's also remember the tail.

```cpp
class DoublyLinkedList
{
private:

    Node* head;

    Node* tail;

public:

    DoublyLinkedList();

    ~DoublyLinkedList();

    void insertFront(int);

    void insertBack(int);

    void remove(int);

    bool search(int);

    void displayForward();

    void displayBackward();

    void clear();
};
```

Mentor:

> "Good engineers don't just store data. They store information that reduces future work."

Keeping the tail saves time.


# Step 7 — Constructor

A new list has nothing.

```cpp
head = nullptr;

tail = nullptr;
```

Simple.


# Step 8 — Reverse Engineer Insert at End

Think.

There are only two situations.

## Case 1

List is empty.

```text
Head

↓

NULL
```

Insert 10.

```text
Head        Tail

 ↓            ↓

+----+----+----+
|NULL| 10 |NULL|
+----+----+----+
```

Head and Tail point to the same node.


## Case 2

List already has nodes.

Suppose

```text
10 ⇄ 20 ⇄ 30
```

Insert 40.

What changes?

Old Tail's next becomes new node.

New node's previous becomes old Tail.

Tail moves to new node.

That's it.

Only three pointer updates.


# Step 9 — Reverse Engineer Deletion

Students often fear deletion.

Don't memorize it.

Think.

Deleting means:

> Remove one bogie and reconnect the remaining train.

Three cases exist.

## Remove Head

Move Head forward.

New Head's previous becomes NULL.


## Remove Tail

Move Tail backward.

New Tail's next becomes NULL.


## Remove Middle Node

Suppose

```text
10 ⇄ 20 ⇄ 30 ⇄ 40
```

Delete 30.

Before

```text
20 ⇄ 30 ⇄ 40
```

After

```text
20 ⇄ 40
```

Two reconnections.

```text
20.next = 40

40.previous = 20
```

Delete node 30.

Done.

Think about neighbours.

Don't think about code first.


# Step 10 — Display Forward

Exactly like Singly Linked List.

Start from Head.

Walk using next.

```text
Head

↓

10 → 20 → 30 → NULL
```


# Step 11 — Display Backward

This is the superpower of a Doubly Linked List.

Start from Tail.

Walk using previous.

```text
Tail

↓

30 ← 20 ← 10 ← NULL
```

Notice something amazing.

We didn't write any reverse algorithm.

The data structure itself allows reverse traversal.

That is why engineers choose the right data structure.

# From Singly to Doubly — What Changed?

| Singly Linked List                                        | Doubly Linked List                                |
| --------------------------------------------------------- | ------------------------------------------------- |
| One pointer (`next`)                                      | Two pointers (`previous`, `next`)                 |
| Forward traversal only                                    | Forward and backward traversal                    |
| Tail insertion requires traversal (unless tail is stored) | Tail insertion is efficient with a `tail` pointer |
| Less memory                                               | More memory                                       |
| Simpler implementation                                    | More pointer updates                              |
| Suitable for one-way navigation                           | Suitable for two-way navigation                   |


# A Message from Your Transflower Mentor

Many students believe learning a new data structure means learning completely new code.

That is not how engineers think.

A Doubly Linked List is simply an evolution of a Singly Linked List.

We identified a limitation, asked the right question, and improved the design by adding one more pointer.

That is software engineering.

**Remember:**

> Don't memorize insertion or deletion algorithms.

> First understand the relationships between nodes.

When you can explain how the pointers change before and after every operation, writing the code becomes the easiest part.

**Good programmers write code.**

**Great software engineers first understand the design, then let the code naturally emerge from that understanding.**
